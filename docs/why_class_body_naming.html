1What alpha = Param(...) really does

The Python statement

alpha = Param(Normal(0, 1))

is not one operation. It's two, evaluated in this order:

  1. Evaluate the right-hand side. Python calls Param(Normal(0, 1)). A new Param object is constructed in memory. At this moment it's anonymous — it has no idea what it will be assigned to.
  2. Bind the name. The current namespace (a dict) gets a new entry: {"alpha": <that Param>}. The dict knows the name. The Param object still doesn't.
The naming asymmetry:
The namespace (the dict) knows the name. The object only knows itself. Names live in the container, not the contents.

So inside Param.__init__, nothing in the language tells the object "you are going to be called alpha". The Param is constructed in a vacuum, then handed to the namespace.

3Operator overloads run blind

When the line

mu = alpha + beta * x

executes, Python's interpreter does:

  1. Look up "alpha" in the class namespace → get the Param object.
  2. Look up "beta" → get its Param object.
  3. Look up "x" → get its Data object.
  4. Call beta.__mul__(x). The interpreter passes the objects, not their names.
  5. Call alpha.__add__(<mul result>). Same: the objects, not names.

Inside __add__, all the method has is self (the Param object) and other (some Data or BinOp object). It has no access to the strings "alpha", "beta", "x".

So when alpha.__add__(...) builds an expression node, it literally cannot write ParamRef("alpha"). It has to write something like ParamRef(self) — a reference to the Param object itself, not its (unknown) name.

The expression tree therefore captures object identities, not names. Names get attached only later, when something with access to the namespace dict (the @model decorator) translates them.

4Aliasing makes "the name" ambiguous

Even if you tried to give the Param its own .name, Python lets you do this:

alpha = beta = Param(Normal(0, 1))
"alpha"
"beta"
↘   ↙
Param#1
.name = ???

One object, two names in the namespace. Which one is "the" name? There is no answer at the language level — the Param object simply doesn't carry that information.

The UnresolvedSymbol(id) trick sidesteps this entirely: the object has an identity (the id), and the decorator later picks the canonical name from the namespace dict using a single rule (first match wins, or raise on aliasing — whichever the project chooses).

5What UnresolvedSymbol buys us

So the situation at the end of the class body is:

  • Every Param / Data object exists in memory but doesn't know its name.
  • The expression tree mu references those objects somehow.
  • The class namespace dict is the only thing that knows "this object is called alpha".

UnresolvedSymbol is the bridge. Each declaration grabs a fresh, unique id at construction:

@dataclass(frozen=True)
class Param:
    distribution: Distribution
    constraint: Constraint | None = None
    size: ... = None
    symbol: UnresolvedSymbol = field(default_factory=next_symbol, init=False, repr=False)

So a ParamRef(UnresolvedSymbol(7)) embedded in an expression is essentially saying:

"I refer to whichever declaration had the unique symbol id 7. Someone with access to the namespace will figure out my real name later."

The id is meaningful only as an identity tag — it survives operator chaining, dataclass construction, hashing, dict lookups. It carries the object's identity across the gap between "expression built" and "names known".

6Sidebar: doesn't __set_name__ solve this?

Python does have a hook for this. Descriptors that implement __set_name__(owner, name) have their name set automatically by the class machinery after the class body completes:

class Param:
    def __set_name__(self, owner, name):
        self.name = name   # called AFTER class body

So why isn't that enough? Because of when it runs:

  1. Class body starts.
  2. alpha = Param(...) — Param constructed, stored in namespace. Has no name yet.
  3. mu = alpha + beta * x — expression tree built right now, using the Param object as it currently is (nameless).
  4. Class body finishes.
  5. Python calls __set_name__ on each descriptor. Param finally learns it's called "alpha".
Too late. By the time __set_name__ runs, mu's expression tree was already constructed and frozen. Setting .name on the Param object now doesn't propagate into the expression node that captured it earlier — the node already exists with whatever it captured.

You'd still need a pass after __set_name__ that walks the expression trees and rewrites them using the now-known names. Which is… basically the @model decorator. So __set_name__ doesn't avoid the work; it just shifts where the name-discovery happens.

The plan's approach (a unique id at construction + one explicit rewrite in @model) is simpler, doesn't depend on descriptor protocol, and keeps everything in one well-defined function.

7Why a single pass is tricky

You might think: "Once @model runs, the namespace is fully built. Why not walk cls.__dict__ once and do both jobs — register declarations AND rewrite expressions — as we go?"

# Naive one-pass attempt
symbols = {}
for name, value in cls.__dict__.items():
    if isinstance(value, (Param, Data)):
        symbols[value.symbol] = name
    elif is_expr_node(value):
        normalize_expr(value, symbols)
    elif isinstance(value, Observed):
        normalize_observed(value, symbols)

This happens to work in simple cases, but the correctness rests on a quiet assumption: every reference in an expression points to a declaration that appears earlier in cls.__dict__.

Where that assumption gets shaky

Case A — Hierarchical Param distributions:

alpha_pop   = Param(Normal(0, 1))           # symbol 1
sigma_alpha = Param(Normal(0, 1))           # symbol 2
alpha       = Param(Normal(alpha_pop, sigma_alpha),  # symbol 3
                    size=n_groups)
y           = Observed(Normal(alpha, 1.0))

The alpha Param's distribution field contains references to alpha_pop, sigma_alpha, and n_groups. To normalize that distribution, the symbol map must already contain entries for symbols 1, 2 and n_groups. With a single pass, the code's correctness silently depends on iteration order matching reference order.

Case B — Future refactor risk:

Dict iteration order is insertion order today, but the moment someone:

  • introduces a metaclass that reorders attributes,
  • adds an inheritance hierarchy where parent declarations interleave,
  • or generates classes programmatically with a different attribute order,

the one-pass approach starts silently producing KeyErrors or, worse, references against a stale partial symbol table.

Case C — Cognitive load:

In a single pass you're interleaving two different intents — "discover declarations" and "rewrite expressions". A reader has to hold both in their head and convince themselves the order works. The bug surface is invisible: if it breaks, it breaks on a model that happens to have an unusual attribute order, and the failure is a KeyError on a symbol that hasn't been registered yet.

The fragility isn't "this will break today". It's "this might break later, in a way that's hard to predict, because correctness depends on something the code never states out loud."

8Two passes spell it out

The two-pass decorator makes the dependency explicit. Pass 1 has one job: build the complete map.

# Pass 1: discover
symbols = {}
for name, value in cls.__dict__.items():
    if isinstance(value, (Param, Data)):
        symbols[value.symbol] = name

After Pass 1, the invariant is loud and clear: every declaration's symbol is in the map. Pass 2 can now rewrite anything in any order:

# Pass 2: rewrite
for name, value in cls.__dict__.items():
    if is_expr_node(value):
        expressions[name] = normalize_expr(value, symbols)
    elif isinstance(value, Observed):
        observed = normalize_observed(value, symbols)
Two-pass: correctness independent of iteration order. Easy to read. Easy to extend.
One-pass: correctness rests on a quiet ordering assumption. Works today, brittle tomorrow.

That's the trade — pay one extra dict iteration (cheap) for a guarantee that future changes don't introduce subtle ordering bugs.

Mental shortcut. Whenever an algorithm has both "discovery" and "use", separating them into discrete passes almost always wins on readability and robustness, even when ordering would technically allow interleaving.
