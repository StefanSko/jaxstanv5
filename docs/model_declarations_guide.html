1The DSL Goal

We want a Stan-like Python DSL where a model is just a decorated class:

@model
class LinearRegression:
    alpha = Param(Normal(0, 1))
    beta  = Param(Normal(0, 1))
    sigma = Param(Normal(0, 1), constraint=Positive())
    x     = Data()
    mu    = alpha + beta * x
    y     = Observed(Normal(mu, sigma))

And it must also support hierarchical priors:

n_groups = Data()
alpha = Param(Normal(alpha_pop, sigma_alpha), size=n_groups)
The trick: mu = alpha + beta * x runs immediately during class-body execution. So Param and Data must behave like symbolic expression objects, not just declarations.

2The Core Challenge

Inside a Python class body, statements execute top-to-bottom in real time. When the interpreter reaches:

mu = alpha + beta * x

…the attributes alpha, beta, x are still just local names with no idea what they'll be called from the outside. The class hasn't finished being built yet.

Problem: Declarations cannot know their final attribute names during class-body execution.

The next several sections unpack why that's true and what it forces us to do.

3What alpha = Param(...) really does

The Python statement

alpha = Param(Normal(0, 1))

is not one operation. It's two, evaluated in this order:

  1. Evaluate the right-hand side. Python calls Param(Normal(0, 1)). A new Param object is constructed in memory. At this moment it's anonymous — it has no idea what it will be assigned to.
  2. Bind the name. The current namespace (a dict) gets a new entry: {"alpha": <that Param>}. The dict knows the name. The Param object still doesn't.
The naming asymmetry:
The namespace (the dict) knows the name. The object only knows itself. Names live in the container, not the contents.

So inside Param.__init__, nothing in the language tells the object "you are going to be called alpha". The Param is constructed in a vacuum, then handed to the namespace.

5Operator overloads run blind

When the line

mu = alpha + beta * x

executes, Python's interpreter does:

  1. Look up "alpha" in the class namespace → get the Param object.
  2. Look up "beta" → get its Param object.
  3. Look up "x" → get its Data object.
  4. Call beta.__mul__(x). The interpreter passes the objects, not their names.
  5. Call alpha.__add__(<mul result>). Same: the objects, not names.

Inside __add__, all the method has is self (the Param object) and other (some Data or BinOp object). It has no access to the strings "alpha", "beta", "x".

So when alpha.__add__(...) builds an expression node, it literally cannot write ParamRef("alpha"). It has to write something like ParamRef(self) — a reference to the Param object itself, not its (unknown) name.

The expression tree therefore captures object identities, not names. Names get attached only later, when something with access to the namespace dict (the @model decorator) translates them.

6Aliasing makes "the name" ambiguous

Even if you tried to give the Param its own .name, Python lets you do this:

alpha = beta = Param(Normal(0, 1))
"alpha"
"beta"
↘   ↙
Param#1
.name = ???

One object, two names in the namespace. Which one is "the" name? There is no answer at the language level — the Param object simply doesn't carry that information.

The UnresolvedSymbol(id) trick sidesteps this entirely: the object has an identity (the id), and the decorator later picks the canonical name from the namespace dict using a single rule.

7The two-step solution: UnresolvedSymbol

The situation at the end of the class body is:

  • Every Param / Data object exists in memory but doesn't know its name.
  • The expression tree references those objects somehow.
  • The class namespace dict is the only thing that knows "this object is called alpha".

UnresolvedSymbol is the bridge. Each declaration grabs a fresh, unique id at construction:

@dataclass(frozen=True)
class UnresolvedSymbol:
    id: int

type Symbol = str | UnresolvedSymbol

A ParamRef(UnresolvedSymbol(7)) embedded in an expression essentially says:

"I refer to whichever declaration had the unique symbol id 7. Someone with access to the namespace will figure out my real name later."
Step 1 · Class body runs
Each Param() / Data() tags itself with a unique UnresolvedSymbol(id). Expressions like alpha + beta * x build a tree of expression nodes that reference these symbols by id.
▼
Step 2 · @model decorator runs
The decorator walks cls.__dict__, learns each declaration's real attribute name, then rewrites every UnresolvedSymbol into the proper string like "alpha".

The lifecycle of a single reference:

Inside class body
ParamRef(UnresolvedSymbol(0))
DataRef(UnresolvedSymbol(1))
▼ @model decorator
After decoration
ParamRef("alpha")
DataRef("x")
Why not a bare int? It works, but UnresolvedSymbol communicates intent and prevents accidental confusion with ordinary numeric constants like ConstNode(0).
Expression nodes (in expr.py) must not know about Param / Data / @model. They only define the symbol type. Keep layers clean.
This is not a monad. It's a deferred symbolic expression graph with a single terminal operation: @model resolves symbols → names.

8Sidebar: doesn't __set_name__ solve this?

Python does have a hook for this. Descriptors that implement __set_name__(owner, name) have their name set automatically by the class machinery after the class body completes:

class Param:
    def __set_name__(self, owner, name):
        self.name = name   # called AFTER class body

So why isn't that enough? Because of when it runs:

  1. Class body starts.
  2. alpha = Param(...) — Param constructed, stored in namespace. Has no name yet.
  3. mu = alpha + beta * x — expression tree built right now, using the Param object as it currently is (nameless).
  4. Class body finishes.
  5. Python calls __set_name__ on each descriptor. Param finally learns it's called "alpha".
Too late. By the time __set_name__ runs, mu's expression tree was already constructed and frozen. Setting .name on the Param object now doesn't propagate into the expression node that captured it earlier — the node already exists with whatever it captured.

You'd still need a pass after __set_name__ that walks the expression trees and rewrites them using the now-known names. Which is… basically the @model decorator. So __set_name__ doesn't avoid the work; it just shifts where the name-discovery happens.

9Declarations vs References

Two concepts that look similar but stay separate:

Declarations

Param, Data, Observed — what you write in the class body.

Expression Refs

ParamRef, DataRef — nodes that appear inside expression trees.

To bridge the two, each declaration exposes .ref() and forwards arithmetic operators:

def __add__(self, other):
    return self.ref() + other

Operators implemented on declarations:

+ - * / reverse versions [ ]

Do not merge Param and ParamRef. One is a node in the metadata; the other is a node in expression trees. Mixing them collapses the two-layer design.

10Why a single pass is tricky

You might think: "Once @model runs, the namespace is fully built. Why not walk cls.__dict__ once and do both jobs — register declarations AND rewrite expressions — as we go?"

# Naive one-pass attempt
symbols = {}
for name, value in cls.__dict__.items():
    if isinstance(value, (Param, Data)):
        symbols[value.symbol] = name
    elif is_expr_node(value):
        normalize_expr(value, symbols)
    elif isinstance(value, Observed):
        normalize_observed(value, symbols)

This happens to work in simple cases, but the correctness rests on a quiet assumption: every reference in an expression points to a declaration that appears earlier in cls.__dict__.

Where that assumption gets shaky

Case A — Hierarchical Param distributions:

alpha_pop   = Param(Normal(0, 1))           # symbol 1
sigma_alpha = Param(Normal(0, 1))           # symbol 2
alpha       = Param(Normal(alpha_pop, sigma_alpha),  # symbol 3
                    size=n_groups)
y           = Observed(Normal(alpha, 1.0))

The alpha Param's distribution field contains references to alpha_pop, sigma_alpha, and n_groups. To normalize that distribution, the symbol map must already contain entries for symbols 1, 2 and n_groups. With a single pass, the code's correctness silently depends on iteration order matching reference order.

Case B — Future refactor risk:

Dict iteration order is insertion order today, but the moment someone:

  • introduces a metaclass that reorders attributes,
  • adds an inheritance hierarchy where parent declarations interleave,
  • or generates classes programmatically with a different attribute order,

the one-pass approach starts silently producing KeyErrors or, worse, references against a stale partial symbol table.

Case C — Cognitive load:

In a single pass you're interleaving two different intents — "discover declarations" and "rewrite expressions". A reader has to hold both in their head and convince themselves the order works. The bug surface is invisible: if it breaks, it breaks on a model that happens to have an unusual attribute order, and the failure is a KeyError on a symbol that hasn't been registered yet.

The fragility isn't "this will break today". It's "this might break later, in a way that's hard to predict, because correctness depends on something the code never states out loud."

11The two-pass decorator

The two-pass decorator makes the dependency explicit. Pass 1 has one job: build the complete map.

# Pass 1: discover
params: dict[str, Param]    = {}
data_slots: list[str]      = []
symbols: dict[UnresolvedSymbol, str] = {}

for name, value in cls.__dict__.items():
    if isinstance(value, Param):
        params[name] = value
        symbols[value.symbol] = name
    elif isinstance(value, Data):
        data_slots.append(name)
        symbols[value.symbol] = name

After Pass 1, the invariant is loud and clear: every declaration's symbol is in the map. Pass 2 can now rewrite anything in any order:

# Pass 2: rewrite
for name, value in cls.__dict__.items():
    if isinstance(value, Observed):
        observed_name = name
        observed = normalize_observed(value, symbols)
    elif is_expr_node(value):
        expressions[name] = normalize_expr(value, symbols)

Resulting metadata

@dataclass(frozen=True)
class ModelMeta:
    params: dict[str, Param]
    data_slots: list[str]
    observed_name: str
    observed: Observed
    expressions: dict[str, ExprNode]
Two-pass: correctness independent of iteration order. Easy to read. Easy to extend.
One-pass: correctness rests on a quiet ordering assumption. Works today, brittle tomorrow.
Validation rules: exactly one Observed; no missing observed variable; no duplicates.
Mental shortcut. Whenever an algorithm has both "discovery" and "use", separating them into discrete passes almost always wins on readability and robustness, even when ordering would technically allow interleaving.

12Symbol Normalization

The decorator must rewrite every UnresolvedSymbol into the real string name across three places:

  1. Free-standing expressions (mu = alpha + beta * x)
  2. Distribution fields (Normal(mu, sigma) on the observed)
  3. Hierarchical prior fields (Normal(alpha_pop, sigma_alpha)) and size= on a Param

Before

BinOp(
  "+",
  ParamRef(UnresolvedSymbol(0)),
  BinOp("*",
    ParamRef(UnresolvedSymbol(1)),
    DataRef(UnresolvedSymbol(2))))

After

BinOp("+",
  ParamRef("alpha"),
  BinOp("*",
    ParamRef("beta"),
    DataRef("x")))

The recursive shape

def normalize_expr(expr, symbols):
    match expr:
        case ParamRef(name) if isinstance(name, UnresolvedSymbol):
            return ParamRef(symbols[name])
        case DataRef(name) if isinstance(name, UnresolvedSymbol):
            return DataRef(symbols[name])
        case ConstNode():
            return expr
        case BinOp(op, l, r):
            return BinOp(op, normalize_expr(l, symbols), normalize_expr(r, symbols))
        case IndexOp(base, idx):
            return IndexOp(normalize_expr(base, symbols), normalize_expr(idx, symbols))

Hierarchical priors need it too

A Param whose distribution contains other Params also gets normalized. So does the size= hint:

Before

Normal(loc=Param(...), scale=Param(...))
size=Data(...)

After

Normal(loc=ParamRef("alpha_pop"), scale=ParamRef("sigma_alpha"))
size=DataRef("n_groups")

13bind() and BoundModel

After decoration, the class gets a bind classmethod. Calling it pairs concrete data with the model and computes parameter shapes.

@dataclass(frozen=True)
class BoundModel:
    meta: ModelMeta
    data: dict[str, jax.Array]
    param_shapes: dict[str, tuple[int, ...]]
    n_params: int

What bind does

  1. Require all explicit data slots.
  2. Require the observed data (by observed_name).
  3. Reject extras.
  4. Convert everything to JAX arrays.
  5. Resolve parameter shapes from size=.
  6. Return BoundModel.

Shape resolution rules

if param.size is None:
    shape = ()
elif isinstance(param.size, int):
    shape = (param.size,)
elif isinstance(param.size, DataRef):
    shape = (int(bound_data[param.size.name]),)

Hierarchical example

bound = HierarchicalRegression.bind(
    n_groups=3,
    y=jnp.asarray([0.0, 1.0, 2.0]))

bound.param_shapes["alpha_pop"]   == ()
bound.param_shapes["sigma_alpha"] == ()
bound.param_shapes["alpha"]       == (3,)
bound.n_params == 5
data_slots contains only explicit Data() declarations; observed_name is kept separate but bind still requires the observed variable to be passed in.
BoundModel holds data + shapes only. No compiler, no sampler, no BlackJAX. That logic lives elsewhere.

14Design Invariants

Rules that must remain true across the whole feature:

Param and Data are declarations.
ParamRef and DataRef are symbolic references.
Class-body expressions use UnresolvedSymbol values.
The @model decorator normalizes UnresolvedSymbol values into real names.
Observed is not a Data slot, but bind(...) still requires observed data.
BoundModel contains data and shapes only; no inference logic.
No BlackJAX in model code.
No compiler logic in model code.
Prefer immutable dataclasses.
Do not use Any.
