1Why measurement error models look different

In an ordinary regression, the observed outcome y is a noisy function of parameters and predictor data:

y ~ Normal(α + β·x, σ)

In a measurement error model, one of the things you observe is itself a noisy version of a latent true value — and that latent true value is the thing the rest of the model actually depends on. Schematically:

y_obs ~ Normal(y_true, y_sd) # observation model
y_true ~ Normal(α + β·x, σ) # latent process model

The observed node (y_obs) is a descendant of a parameter (y_true) — exactly the question you asked.

Two distinct likelihoods get added to the log posterior: one for "the measurement was noisy" and one for "the latent truth followed this process". The DSL has to give us a way to express both.

2McElreath model 15.1 — divorce with outcome error

Straight from the R script (single measurement error, on the outcome only):

m15.1 <- ulam(
  alist(
    D_obs ~ dnorm( D_true , D_sd ),
    vector[N]:D_true ~ dnorm( mu , sigma ),
    mu <- a + bA*A + bM*M,
    a     ~ dnorm(0, 0.2),
    bA    ~ dnorm(0, 0.5),
    bM    ~ dnorm(0, 0.5),
    sigma ~ dexp(1)
  ),
  data=dlist, chains=4, cores=4)

In math notation:

D_obsi ~ Normal(D_truei, D_sdi)
D_truei ~ Normal(μi, σ)
μi = a + bA·Ai + bM·Mi
a ~ Normal(0, 0.2)
bA ~ Normal(0, 0.5)
bM ~ Normal(0, 0.5)
σ ~ Exponential(1)

Key actors:

Parameters

a, bA, bM, σ (scalars) and D_true (vector of length N)

Data

A, M, D_sd, N — all known up front

Observed

D_obs — the noisy divorce-rate measurement

Deterministic

μ — pure expression of params + data

4Does the current DSL handle this?

Let's try writing model 15.1 in the planned @model DSL. The key trick: D_true is a Param whose distribution depends on other params — exactly the hierarchical pattern already shown in the implementation plan.

@model
class DivorceMeasErr:
    # data
    N    = Data()
    A    = Data()
    M    = Data()
    D_sd = Data()

    # scalar params (priors as in m15.1)
    a     = Param(Normal(0.0, 0.2))
    bA    = Param(Normal(0.0, 0.5))
    bM    = Param(Normal(0.0, 0.5))
    sigma = Param(Exponential(1.0), constraint=Positive())

    # latent linear predictor (a deterministic expression)
    mu = a + bA * A + bM * M

    # latent true outcome — a hierarchical Param (Normal centered on mu)
    D_true = Param(Normal(mu, sigma), size=N)

    # the single observed quantity
    D_obs = Observed(Normal(D_true, D_sd))
Yes Model 15.1 fits the current DSL as-is. It needs only one Observed (D_obs). The latent D_true is a hierarchical Param. The expression mu and the conditional distribution Normal(mu, sigma) are exactly what the plan already supports.

5McElreath model 15.2 — error on both sides

Now both divorce and marriage are measured with error:

m15.2 <- ulam(
  alist(
    D_obs ~ dnorm( D_true , D_sd ),
    vector[N]:D_true ~ dnorm( mu , sigma ),
    mu <- a + bA*A + bM*M_true[i],
    a     ~ dnorm(0, 0.2),
    bA    ~ dnorm(0, 0.5),
    bM    ~ dnorm(0, 0.5),
    sigma ~ dexp(1),

    M_obs ~ dnorm( M_true , M_sd ),
    vector[N]:M_true ~ dnorm( nu , tau ),
    nu  <- aM + bAM*A,
    aM  ~ dnorm(0, 0.2),
    bAM ~ dnorm(0, 0.5),
    tau ~ dexp(1)
  ), data=dlist2, chains=4, cores=4)

Now there are two observation models:

D_obsi ~ Normal(D_truei, D_sdi)
M_obsi ~ Normal(M_truei, M_sdi)

And two latent process models, with the latent M_true entering the linear predictor for D_true.

6m15.2 as a DAG

Two green observed nodes — M_obs and D_obs — both descend from latent parameters. The cross-link M_true → μ (dashed purple) is the whole point: the regression uses the latent marriage rate, not the noisy measurement.

7The wall: one Observed isn't enough

The plan states this design invariant:

Validation: exactly one Observed; no missing observed variable; no duplicate observed declaration.

That works for ordinary regression, and it works for m15.1 (single measurement error). But m15.2 has two distinct observation likelihoods that both need to contribute to the joint log-density:

log p = Σi log N(D_obsi | D_truei, D_sdi) + Σi log N(M_obsi | M_truei, M_sdi) + (priors on params)
No Model 15.2 cannot be expressed with the current DSL. The "exactly one Observed" rule blocks the second observation likelihood. We can't smuggle M_obs in as a Data slot either, because Data carries no distribution — it would contribute nothing to the log posterior.

8A minimal extension: many Observed

The cleanest fix: relax the "exactly one Observed" rule to "one or more". Most of the plan stays the same; ModelMeta just stores a dict instead of a single name.

Before

@dataclass(frozen=True)
class ModelMeta:
    params: dict[str, Param]
    data_slots: list[str]
    observed_name: str
    observed: Observed
    expressions: dict[str, ExprNode]

After

@dataclass(frozen=True)
class ModelMeta:
    params: dict[str, Param]
    data_slots: list[str]
    observed: dict[str, Observed]      # >=1 entry
    expressions: dict[str, ExprNode]

Validation changes are tiny:

  • len(observed) >= 1 — at least one observation likelihood.
  • No duplicate names (already enforced by class attribute uniqueness).
  • No overlap between an Observed name and a Data slot or Param name.

And the decorator's Pass 2 collects them into the dict instead of asserting one:

for name, value in cls.__dict__.items():
    if isinstance(value, Observed):
        observed[name] = normalize_observed(value, symbols)
    elif is_expr_node(value):
        expressions[name] = normalize_expr(value, symbols)
Nothing about Param, Data, UnresolvedSymbol, the two-pass decorator, or expression normalization needs to change. The extension is local.

9m15.2 in the extended DSL

With the multi-Observed relaxation, m15.2 becomes a direct translation of the R code:

@model
class DivorceMarriageError:
    # --- data ---
    N    = Data()
    A    = Data()
    D_sd = Data()
    M_sd = Data()

    # --- marriage sub-model ---
    aM    = Param(Normal(0.0, 0.2))
    bAM   = Param(Normal(0.0, 0.5))
    tau   = Param(Exponential(1.0), constraint=Positive())
    nu    = aM + bAM * A
    M_true = Param(Normal(nu, tau), size=N)

    # --- divorce sub-model ---
    a     = Param(Normal(0.0, 0.2))
    bA    = Param(Normal(0.0, 0.5))
    bM    = Param(Normal(0.0, 0.5))
    sigma = Param(Exponential(1.0), constraint=Positive())
    mu    = a + bA * A + bM * M_true       # uses LATENT M_true
    D_true = Param(Normal(mu, sigma), size=N)

    # --- two observation likelihoods ---
    M_obs = Observed(Normal(M_true, M_sd))
    D_obs = Observed(Normal(D_true, D_sd))

The expression nodes nu and mu reference Params and Params-that-are-themselves-vectors (M_true). Everything else — symbol resolution, hierarchical sizing, the recursive normalization for BinOp / ParamRef / DataRef — already works.

Author ordering note. Python's class body executes top-to-bottom, so M_true must be defined before mu references it. That's a natural ordering anyway (latent process → linear predictor → outcome).

10What changes in bind and the log-density

bind()

Today, expected = set(meta.data_slots + [meta.observed_name]). With many Observed, the union just widens:

expected = set(meta.data_slots) | set(meta.observed.keys())

For m15.2 that would be {N, A, D_sd, M_sd, D_obs, M_obs}. bind still requires the full set, rejects extras, and converts each value to a JAX array — same logic, slightly wider expectation set.

Log-density (downstream of model code)

The plan keeps inference logic outside model/, but here's the conceptual picture for the compiler that consumes BoundModel:

log p(θ | data) =   Σp ∈ params log π(θp | parents(p))
  + Σo ∈ observed log L(data[o] | θ, data)

With one Observed, the second sum has one term. With many, it has many. The compiler just walks meta.observed and adds each contribution to the log-density. No new machinery — just a loop where there used to be a single term.

Latent params like D_true and M_true contribute via the prior sum (their conditional priors Normal(mu, sigma), Normal(nu, tau)). Observed nodes contribute via the likelihood sum. The DAG decides which sum each node belongs to.

11Summary

As-is m15.1 — single measurement error works in the current DSL. The latent D_true is a hierarchical Param whose distribution is centered on an expression of other Params/Data. One Observed suffices.
Needs ext. m15.2 — dual measurement error needs two observation likelihoods. The "exactly one Observed" invariant must be relaxed to "one or more". All other plan invariants stand.

The diff in plain words

  • DSL surface: exactly the same — users still write name = Observed(distribution). They just may write more than one.
  • ModelMeta: observed: Observed + observed_name: str → observed: dict[str, Observed].
  • Decorator: Pass 2 collects all Observed into a dict instead of asserting uniqueness.
  • Validation: require len(observed) >= 1 instead of exactly one.
  • bind: expected data names = data_slots ∪ observed names.
  • Downstream compiler: sum log-likelihood contributions across all observed nodes instead of one.
Importantly, the cleanest test for the extension is m15.1 first (still single-Observed, exercises latent hierarchical Params + observed-as-descendant-of-param), then m15.2 to validate the multi-Observed surface. That mirrors McElreath's own pedagogical order.
